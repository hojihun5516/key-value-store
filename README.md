# 가상 면접이라고 가정하고 요구사항을 정리
```
A: K,V스토어를 만들어주세요

Q: 이 스토어는 어떤 서비스에 사용될 용도인가요?
A: 게시글 혹은 제품의 리뷰 서비스를 위해 사용될 예정입니다

Q: 하루에 예상되는 저장할 데이터 요청은 어느정도인가요?
A: 하루에 1천건의 쓰기 요청을 예상하고있습니다. 하지만 확장성을 고려했으면 합니다

Q: 하루에 예상되는 읽기 요청은 어느정도인가요?
A: DAU 4만으로 한명의 유저당 3개의 글을 조회해서 리뷰를 읽어온다고 가정하여 12만번의 읽기 요청이 예상됩니다.

Q: 리뷰 서비스 특성상 장애가 발생했을 때 몇개의 데이터를 보여주지 못하더라도 장애처럼 보이지 않게 하는게 중요할것 처럼 보이는데 맞나요?
A: 네 저장소에 장애가 생겼을 경우에 장애처럼 보이지 않게 하는게 중요합니다

Q: 파티셔닝 된 저장소에 장애가 발생했을 때 백업, 삭제, 추가 에 대해 고려해봐야 하는데 완벽한 서비스를 구현하려면 많은 시간이 걸릴 것 같습니다
우선순위를 부탁드립니다
A: 네 구현에 대한 시간관계상 파티셔닝 된 저장소에 문제가 발생했다면 삭제 -> 추가 -> 복원 순서대로 우선순위를 지정하겠습니다.

Q: 분산 처리를 위해서는 Service Discovery라는 개념이 필요합니다.
메인 요구사항인 K,V스토어 구현에 초점을 맞춰서 저장소의 물리적 위치는 Service Registry에 인메모리로 저장해도 될까요?
A: 네 괜찮습니다

Q: 저장할 때 파일시스템에 접근하는 방법과 애플리케이션 인메모리에 저장하는 방식이 있습니다.
복원을 고려한다면 파일 시스템에 저장해야겠지만 설계 요구사항에 가장 마지막인 복원 보다는
주어진 시간안에 구현해야 하기 때문에 애플리케이션 인메모리에 저장 해도 될까요?
A: 네 저장 방법은 구현 후에 변경 가능 하기 때문에 시간 안에 완성하는 것을 목표로 하는게 좋겠어요
```
